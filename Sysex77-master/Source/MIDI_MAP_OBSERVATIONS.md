# SY99 Pan EG — SysEx Mapping Observations

Журнал строится на **ваших post-fix дампах** из монитора **Sysex77** (после исправления очереди в **MidiDemo.h**): один спокойный шаг по **одному** числовому полю → **одна строка** лога с одним префиксом адреса и одним **`byte[8]`**.

### ⚠ Ограничения: баг буфера и «ложные» строки в журнале

Ранее в мониторе из‑за **ошибки очереди / дублирования буфера** в **MidiDemo** в один и тот же лог могли попадать **повторы и перепутанный порядок** сообщений. Поэтому **не всё содержимое этого файла равнозначно надёжно**.

**Считать опорными (можно использовать для карты параметров и CSV):**

- раздел **CONFIRMED Parameters**;
- явные пометки **post-fix RECORDED** и монотонные лестницы **±1** (**`[#1]`…`[#63]`** или полная **0…127** там, где это описано);
- строки таблицы UNKNOWN с **✅** и привязкой к таким лестницам;
- канон Pan EG (**`byte[3]=0x0A`**) по вкладкам **Rate / Level** после фикса монитора.

**Относить к гипотезам / не импортировать в код без повторной проверки на железе:**

- пометки **legacy**, **архив**, **PROBABLE**, **❓**, короткие **фрагменты** без чистой лестницы;
- композитные дампы, где один жест смешивает **Pan**, **AWM** и старые адреса (**`03`/`07`** vs **`0A`/`03`** и т.п.);
- любые строки из логов **до** исправления **`batch` / очереди** в **MidiDemo.h** (см. таблицу достоверности ниже).

При переносе адресов во внешние таблицы (**`_agent_context/*.csv`**) брать **только** явно подтверждённые блоки, а не сам файл целиком.

**О нумерации `[#N]`:** **`[#1]` — первая строка данной вставки текста**; между разными параметрами номера **не сквозные**.

### Лестницы **`[#1]`…`[#63]`** (ровно 63 строки)

Это **ровно 63 последовательных изменения** значения (**±1** при правильном INC/DEC). Именно такие серии (**по 63 строки на параметр**) считаются здесь **точными опорными дампами**. Жёсткая привязка к шкале:

- **Rate** (HT, R1–R4, RR1, RR2): в логах зафиксировано **`byte[8]` = `01`…`3F`** — на экране **1…63**. **Показание 0** (и кадр **`byte[8]=0x00`**) в этих 63 строках **нет** — **не указано**, потому что **до `[#1]`** синт уже мог стоять на **0** без отдельного SysEx или первый зафиксированный шаг был **0→1** без строки для **0**. Полный домен по железу и таблице канона всё равно **0…63**.
- **Level** (L0–L4, RL1, RL2): те же **63 строки** — эталонная развёртка шкалы **−32…+31**. Край **−32** (**`raw=0x00`**) в этих строках **часто не указан** по той же причине, что и **Rate 0**: только **до `[#1]`**, без отдельного кадра.

**Кратко, что именно «пропало» в типичном дампе:** **Rate — 0**, **Level — −32**, на MIDI это один и тот же **`byte[8]=0x00`** — разный смысл только после правила **`disp ≈ raw − 0x20`** для Level.

**Кодирование** (не путается с «дыркой» в логе):

- **Rate:** **`byte[8]`** — **прямое** значение **0…63**.
- **Level:** **`byte[8]`** = **`raw`** **`0x00…0x3F`**, **`display ≈ (int)raw − 0x20`** → **−32…+31**.

Явная проверка **`raw = 0x00`** (Rate **0** или Level **−32**) — один лишний шаг от минимума / старт «с нуля», чтобы в монитор попала отдельная строка.

### Достоверность монитора

| Источник | Как использовать |
|----------|----------------|
| Лог **до** фикса **MidiDemo** | не канон для карты параметров; возможны **дубликаты строк** и неверный порядок из‑за бага буфера |
| Лог **после** фикса | канон для таблицы ниже *только* там, где есть явное **RECORDED** / лестница; опционально **`[MidiMon]`** (`SYSEX77_MIDI_MONITOR_DIAGNOSTICS`) |

На один щелчок несколько строк — проверять второй MIDI‑вход, Thru/loopback, ускорение энкодера.

## SysEx Message Format (9 bytes between F0 and F7)

```
F0  43  1n  34  [b3]  [b4]  [b5]  [b6]  [b7]  [b8]  F7
     |       |    |     |     |     |     |     |
  Yamaha  model block elem  sub  param  MSB  value
                      off
```

| Index | Name | Notes |
|-------|------|-------|
| [0] | `0x43` | Yamaha |
| [1] | `0x10` | Device ID (channel 0) |
| [2] | `0x34` | SY77/99 Voice param |
| [3] | block | Parameter block type (varies by param!) |
| [4] | elem offset | `0x00`=E1, `0x20`=E2, `0x40`=E3, `0x60`=E4 |
| [5] | sub | Usually `0x00` |
| [6] | addr | Parameter address within block |
| [7] | `0x00` | Data MSB (always 0 so far) |
| [8] | value | Data LSB (actual value) |

### Правило адресации

Сначала фиксируют **`byte[3]`** — какой это **блок** параметров. **`byte[6]`** читают **только внутри этого блока**. Совпадение **`byte[6]`** при разном **`byte[3]`** — норма.

В композитных логах без контекста экрана часто «рядом» с Pan встречаются: **`43 10 34 07 00 00 50`** — **AWM EG R2 (E1)**; **`43 10 34 07 40 00 51`** — **AWM EG R2 (E3)**; **`byte[3]=0x07`, `byte[6]=0x06`** — legacy **Hold**, см. **Pan EG HT** (не то же самое, что **Pan Rate R4** при **`0x0A`/`06`**).

### Pan EG — **один канон** (элемент **E1**, `byte[4]=0x00`; для **E2/E3** тот же **`byte[6]`**, заменить **`byte[4]`** на **`20`/`40`**)

Кадр: **`F0 43 10 34 0A [byte4] 00 [byte6] 00 [byte8] F7`**.

| Параметр | Вкладка | `byte[6]` | Значение `byte[8]` | Диапазон на SY99 |
|----------|---------|-----------|-------------------|------------------|
| **HT** | Rate | `0x02` | прямое `raw` | **0…63** |
| **R1** | Rate | `0x03` | прямое `raw` | **0…63** |
| **R2** | Rate | `0x04` | прямое `raw` | **0…63** |
| **R3** | Rate | `0x05` | прямое `raw` | **0…63** |
| **R4** | Rate | `0x06` | прямое `raw` | **0…63** |
| **RR1** | Rate | `0x07` | прямое `raw` | **0…63** |
| **RR2** | Rate | `0x08` | прямое `raw` | **0…63** |
| **L0** | Level | `0x09` | `raw`; **`disp ≈ raw − 0x20`** | **−32…+31** |
| **L1** | Level | `0x0A` | то же | **−32…+31** |
| **L2** | Level | `0x0B` | то же | **−32…+31** |
| **L3** | Level | `0x0C` | то же | **−32…+31** |
| **L4** | Level | `0x0D` | то же | **−32…+31** |
| **RL1** | Level | `0x0E` | то же | **−32…+31** |
| **RL2** | Level | `0x0F` | то же | **−32…+31** |
| **SLP** | Level | `0x10` | прямое `raw` | **`s1…s4` ↔ `0…3`** |

**Редактор `Pan.h`:** отправка Level/Rate для Pan EG использует **`byte[3]=0x0A`** и **`byte[6]`** из таблицы (**`attachMidiControllers`**).

### Element Pan — Common mode / F8 Edit (тот же блок **`byte[3]=0x0A`**, **`byte[4]`** по элементу как выше)

Захват: **Common → Element pan → F8 edit**.

| Параметр | UI | `byte[6]` | `byte[8]` | Примечание |
|----------|-----|-----------|-----------|------------|
| **Pan Source** | Velocity / Note number / LFO | **`0x00`** | **`0`** / **`1`** / **`2`** | **`43 10 34 0A 00 00 00 00 xx`** |
| **Source Depth** | глубина модуляции | **`0x01`** | **0…127** (**`0x00…0x7F`**) | **`43 10 34 0A 00 00 01 00 xx`** |
| **03 Name** | имя | — | — | Пункт меню **есть**; SysEx строки имени **не разбирали**. **`«03»`** — номер в **F8 edit**, не синоним **`byte[6]`** без дампа. |

**Source Depth — кодирование (RECORDED):** адрес **`0x0A`/`01`**. **`byte[8]`** — **прямое линейное** значение **0…127** (**`0x00…0x7F`**), без особого bias.

**Подтверждение полной монотонности:** захват **«127→0»**: **`[#0]`…`[#127]`** — **128** сообщений, **`byte[8]`** подряд **`0x7F`→`0x7E`→…→`0x01`→`0x00`** (**−1** на строку лога). Ранний проход с разрывами по кодам был из‑за **Data Entry** «с проскоками», а не из‑за карты параметров на SY99.

Редактор **`Pan.h`** эти поля пока **не** подвешивает на MIDI — только журнал.

### AFM Operator EG (`byte[3]=0x56`) — RECORDED

Префикс (в дампах ниже **`byte[4]=0x00`**): **`F0 43 10 34 56 [byte4] 00 …`**

#### HT

Меню: **AFM operator EG → HT**, шкала **0…63**. Кадр: **`F0 43 10 34 56 [byte4] 00 0D 00 xx F7`**. **`byte[8]`:** прямое **0…63**. **Evidence:** **`[#0]`…`[#63]`** — спуск **`3F…00`**.

#### R1

Меню: **AFM operator EG → R1**. Кадр: **`F0 43 10 34 56 [byte4] 00 00 00 xx F7`**. **`byte[8]`:** **0…63**. **Evidence:** **`[#0]`…`[#63]`** — подъём **`00…3F`**.

#### R2

Меню: **AFM operator EG → R2**. Кадр: **`F0 43 10 34 56 [byte4] 00 01 00 xx F7`**. **`byte[8]`:** **0…63**. **Evidence:** **`[#0]`…`[#63]`** — подъём **`00…3F`**.

## Critical structural findings

### Legacy (**не** канон вкладок **`byte[3]=0x0A`**)

- **Hold (архив / график):** **`byte[3]=0x07`, `byte[6]=0x06`**.
- **Pan R1 (архив):** **`byte[3]=0x03`, `byte[6]=0x07`** — см. **Pan EG R1**.

### График Pan EG

Движение **общей точки** может давать **два** адреса подряд (legacy **HT+R1**). Для таблицы параметров брать **числовые поля вкладки Rate**.

**`Pan.h`:** независимая отправка **HT** **`0x02`** и **R1** **`0x03`** в блоке **`0x0A`** согласована с записями.

## CONFIRMED Parameters

### AWM EG R2 — **CONFIRMED** (block **`0x07`**)

**Опорная связка:** **`43 10 34 07 40 00 51 00 xx`** — **AWM EG R2**, **E3** (**`byte[4]=0x40`**, **`byte[6]=0x51`**). **`byte[8]`:** прямое **0…63** на экране в монотонных спусках. Рядом возможен **`43 10 34 07 40 00 4F 00`** (**`4F`**, **0–1**) — см. UNKNOWN.

**Частый фон:** **`43 10 34 0A 00 00 04 00`** — это **Pan EG R2** (**Rate**), другой параметр.

---

**AWM EG R2 на E1:** **`byte[3]=0x07`, `byte[4]=0x00`, `byte[6]=0x50`**.
- **Захват («AWM EG 00–62»):** **[#1]–[#63]** — **`xx`** **`01`…`3F`**; **`показание ≈ byte[8] − 1`** (**`0x3F`** ↔ **62**).
- **`07`/`50`** вперемешку в дампах **Pan** без редактирования AWM — строки другого контекста; для **AWM R2 на E3** опора **`07 40 … 51`**.
- Историческая ошибка журнала: **`07`/`50`** помечали как «AWM R1» — это **AWM EG R2 (E1)**.

### Pan EG HT (Hold Time)

**Два наблюдаемых адреса (свести одним контрольным проходом):**

| Источник | Адрес SysEx | Примечание |
|----------|-------------|------------|
| Ранние дампы (до бага монитора / смесь сообщений) | **`byte[3]=0x07`, `byte[6]=0x06`** | Часто вместе с **R1** **`03`/`07`** при движении общей точки на графике. |
| **Новый захват, монитор после фикса `batch`** | **`byte[3]=0x0A`, `byte[6]=0x02`** | Подпись пользователя: **Pan EG → Rate**, поле **HT**, шкала **0–63**; см. блок ниже. |

---

#### Новый захват — **monitor post batch fix** (**явное имя параметра:** **HT**, **PAN EG**, вкладка **Rate**)

- **Подтверждение пользователя:** тот же дамп (**параметры по описанию «63→00»**) — это **не другой Rate**, а именно **Hold Time** на **PAN EG → Rate**.
- **Контекст:** экран **PAN EG**, вкладка **Rate**, поле **HT**, монотонное уменьшение (**жест «вниз» по шкале**).
- **Кадр:** `F0 43 10 34 0A 00 00 02 00 xx F7` (**E1:** `byte[4]=0x00`).
- **Evidence:** **`[#1]`–`[#63]`** — **`byte[8]`** подряд **`0x3E`→`0x3D`→…→`0x01`→`0x00`** (шаг **−1** на сообщение). По словам пользователя, на LCD шла цепочка **63→62→61→…→0**; тогда **`byte[8]` совпадает с показанием «после шага»** (первое сообщение **`0x3E`** = **62**, т.е. первый зафиксированный шаг — переход с **63** на **62**). Если бы в лог попадало значение «до шага», на **`[#1]`** ожидался бы **`0x3F`**.
- **Кодирование:** **`byte[8]`** здесь — **прямое линейное** значение в шагах (**без `−0x20`** как у Level).
- **Вывод:** адрес **`0x0A`/`0x02`** — канон для **HT с вкладки Rate**. Альтернатива **`07`/`06`** — см. **Critical structural findings**.

## PROBABLE Parameters (needs clean 5-INC confirmation)

### Pan EG R1 (Rate 1)

#### RECORDED — **PAN EG → Rate**, поле **R1**, шкала **00–63** (monitor **post batch fix**)

- **Подпись пользователя:** **R1 | PAN EG | Rate**, монотонный проход (**по описанию 00→63**).
- **Кадр:** `F0 43 10 34 0A 00 00 03 00 xx F7` (**E1:** `byte[4]=0x00`).
- **Evidence:** **`[#1]`–`[#63]`** — **`byte[8]`** подряд **`0x01`→`0x02`→…→`0x3F`** (**63 сообщения**). Экранные **«00»** в этом дампе не видны как **`byte[8]=0x00`** — если показание **00** соответствует «тишине» на шине или отдельному кадру, имеет смысл один короткий тест с фокусом на **00**.
- **Кодирование (этот проход):** **`byte[8]`** = целое **1…63** (`0x01…0x3F`) при движении вверх по вашему жесту (**прямая линейная лестница** на шаг сообщения).
- **Вывод для редактора:** с вкладки **Rate** числовой **R1** на SY99 уходит в блок **`0x0A`, параметр `byte[6]=0x03`** — **не** в **`0x03`/`0x07`** из архивных дампов.

#### Архив — **R1** как **`byte[3]=0x03`, `byte[6]=0x07`** (старые сессии / другой UI-путь)

- В **нескольких старых** тестах стабильно встречались **`03`/`07`** с **`byte[8]`** вплоть до **`0x5F` (95)** — т.е. **расширенный диапазон** за пределами **0…63**.
- Поведение: смена **R1** могла сопровождаться **рассылкой по элементам** (`byte[4]=00/20/40/…`) — **не ошибка** пользователя, но путь **`0x03`** vs **`0x0A`** для **того же имени «R1»** надо держать открытым до контроля (**график vs число Rate**, другой экран и т.д.).
- Строки вроде **`[#110]`–`[#188]`** с **`03`/`07`** в старых вставках относятся сюда.

### Pan EG R2 (Rate 2)

#### RECORDED — **PAN EG → Rate**, поле **R2**, шкала **00–63** (monitor **post batch fix**)

- **Подпись пользователя:** **R2 | PAN EG | Rate**, монотонный проход (**по описанию 00→63**).
- **Кадр:** `F0 43 10 34 0A 00 00 04 00 xx F7` (**E1:** `byte[4]=0x00`).
- **Evidence:** **`[#1]`–`[#63]`** — **`byte[8]`** подряд **`0x01`→`0x02`→…→`0x3F`** (**63 сообщения**). Как у **R1** на вкладке Rate: в этом дампе **нет** **`byte[8]=0x00`** — при необходимости контрольный шаг на показание **00**.
- **Кодирование (этот проход):** **`byte[8]`** = **1…63** при движении «вверх» по шкале (**линейная лестница** на сообщение).
- **Вывод для редактора:** числовой **R2** с вкладки **Rate** — блок **`0x0A`, `byte[6]=0x04`**.

#### Архив и смесь с **AWM EG R2**

- **`34 7 0 0 50 0`** = **AWM EG R2 (E1)** — см. **AWM EG R2**; строки **`[#1]`–`[#63]`** вида **`50`** в композитном логе **«PAN EG Rate»** — **не** доказательство **Pan** R2.
- **Старый захват с подписью R2 63→0:** в одном файле **`[#1]`–`[#63]`** шли как **`07`/`50`** (AWM); для **Pan** смотрели **`0A`/`04`** и **`03`/`08`**.
- **`[#90]`–`[#94]`:** **`03`/`08`** — короткий фрагмент (**`4,6,4,2,0`**); **не** полная шкала — оставить как **гипотезу другого пути** без имени поля.
- **`[#862]`–`[#925]`:** **`0A`/`04`**, монотонный спуск **`3F … 00`** (**64 шага**) — тот же адрес **Pan EG R2**, направление жеста **вниз** (прямые **`byte[8]`** как **0…63**).

#### Архив — композитный дамп (подпись **R4 00–63**, другая нумерация **[#n]**)

- **`[#991]`–`[#1013]`:** **`0A`/`06`** — **неровная** преамбула; **не** опорная калибровка.
- **`[#1014]`–`[#1077]`:** монотонное **`xx`:** **`00`→`01`→…→`0x3F`** (**64 сообщения**) — полный **`byte[8]`** 0…63 в MIDI; отличается от post-fix файла наличием **`00`** в начале шкалы (как у **R3** в архиве).
- **`[#1078]` …** того же дампа: снова **`0A`/`06`**, последовательность **без** монотонной лестницы — **хвост жеста**.

## Pan EG Level и SLP — детали (= таблица канона выше)

**L0–L4, RL1, RL2**

- Кадр (**E1**, `byte[4]=0x00`): **`F0 43 10 34 0A 00 00 [byte6] 00 xx F7`** — **`[byte6]`** по строкам **`09…0F`** таблицы канона.
- **`byte[8]`:** **`raw`** **`0x00…0x3F`**; **`display ≈ (int)raw − 0x20`** ⇒ **−32…+31** (ровно **64** значения; верх экрана **+31**, не +32).
- Твои лестницы **`[#1]`…`[#63]`** с **`xx = 01…3F`** задают **63 шага подряд**; см. раздел **«Лестницы 63 строки»** в шапке — **начальный `00`** часто **не попадает в лог**, если первый шаг был уже от **`raw = 0x00`**.
- Уровни и номера **`byte[6]`** при **`0x0A`** — только по **главной таблице канона** (вкладки Rate vs Level).

**SLP**

- Кадр (**E1**): **`F0 43 10 34 0A 00 00 10 00 xx F7`** — **`byte[8]`** прямое: **`S1→0`**, **`S2→1`**, **`S3→2`**, **`S4→3`** (без **`−0x20`**).
- **RECORDED (пользователь):** **`[#0]`…`[#3]`** — подряд **`xx = 00`…`03`** при подписях **S1…S4**; пятого значения нет (**не `0…4`**).
- **`Pan.h`:** **`sliderSLP`** — диапазон **`0…3`**, **`byte[8]=raw`**; визуализация EG: **`setLoopPoint(raw + 1)`** (**S1…S4** ↔ скобка **L1…L4**).

**Фильтры Ctrl+F (Level):** `43 10 34 0A 00 00 09 00 ` … `10 00 ` — см. блок **Practical filters** ниже.

## Capture behaviour — OBSERVED (do not confuse with “one knob = one SysEx”)

Users report wildly different **total** line counts (320 vs 390 vs 400+) for “the same edit”. This does **NOT** disprove mapping — total count mixes many independent streams:

1. Один дамп может смешивать **AWM**, **Pan**, legacy — см. **Правило адресации** выше.
2. **Pan EG coupling:** при движении **общей точки** на графике рядом могут идти **legacy HT `07`/`06`** и **R1 `03`/`07`**; при редактировании **числом на вкладке Rate** — **HT `0A`/`02`**, **R1 `0A`/`03`** (post-fix RECORDED).
3. **Multi‑element echoes:** в **legacy**-ветке **`0x03`/`0x07`** смена **R1** могла давать префиксы по элементам (**`byte[4]=00/20/40`**); для **`0x0A`/`0x03`** то же нужно подтвердить отдельным дампом.
4. **Clear resets [#n] labels** only in the editor — SY99 spam volume depends on dwell time / encoder accel / reversing direction mid‑turn → **counts are not reproducible KPIs**.
5. **LCD vs MIDI:** для Level (**`raw−0x20`**) и Rate (**прямой `byte[8]`**) правило одно и то же, что в **таблице канона**; «перегоны» энкодера могут давать пропуски между строками — опора только на **монотонные ±1** лестницы.
6. **«Грязные» участки имеют смысл, но другой задачи.** Рывки, дубли, скачки и «шумная» преамбула **не выбрасываются как мусор абсолютно** — их часто вызывает **реально**: ускорение энкодера (не каждый квант интерфейса → отдельный SysEx), **связанные параметры** (точка на графике дергает сразу два адреса), **рассылка по элементам**, смена фокуса/страницы, фоновые строки (**`0x0A`/`0x02`**, **`0x03`** и т.п.). Это полезно **для понимания поведения железа** и последующего тестирования отправки из редактора, но для таблицы **«один параметр ↔ один закон кодирования»** такие куски **нельзя** брать как опорную калибровку: их обязательно дополняют **монотонной лестницей ±1** (INC/DEC) по **одному** выбранному полю после Clear.
